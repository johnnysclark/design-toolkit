Rhino Wizard

How it works

A Rhino / Grasshopper / GhPython tutor that teaches you the workflow behind a result — and, at Beginner, refuses to just hand you the answer. It grades the trace, not the screenshot.

One page, two halves: how a student uses it · how it's built.

Pick a mode (Rhino commands, Grasshopper components, or GhPython code) and a level (Beginner → Moderate → Advanced). The level decides how much the tutor withholds; the mode decides the vocabulary it speaks. Every factual claim it makes is tagged so you know what to trust and what to check.

PART 1 Using the tutor

1 · Join

Enter the class code your instructor gives you and a name. No password.

2 · Set mode & level

Rhino / Grasshopper / GhPython, and Beginner / Moderate / Advanced.

3 · Ask

Type a question, upload a sketch photo or GH screenshot, or paste a .ghx.

4 · Read & check

You get a structured answer; every claim is tagged ✓ / ? / ⚠.

5 · Report back

Tell it what you observed to unlock the next step. That's your trace.

Why it makes you report back: at Beginner and Moderate the tutor ends with a "try it and tell me what you saw" checkpoint. It won't continue until you report a real observation (a branch count, an error message). No report, no next step — and that back-and-forth becomes the learning record your instructor sees.

PART 1·5 The three ideas behind it

It withholds on purpose

At Beginner the answer format literally has no place to put a finished solution — only a skeleton, the one next step, and the "why." The tutor can't dump the answer because there's nowhere to put it.

Every claim is tagged

The model is a material to interrogate, not an oracle. Each fact carries a confidence tag and a reason, so you verify the rest.

✓ stable ? version-dependent ⚠ likely wrong — check

The trace is the point

Questions, answers, sketches, and your report-backs are all recorded. Success is a rich trace you can re-derive from next week — not a screenshot you can't reproduce.

PART 2 How it's built

A small, transparent stack — one Node server, one Postgres database, and the Claude API. No build step, no framework; the seams are on purpose.

Identity, lightly

Students join with a class code + name and get an unguessable token. No passwords; one student can't read another's data.

Streaming answers

The server streams the model's progress, then sends the final schema-checked answer — so the page feels live but the structure is enforced.

Runs anywhere

One always-on Node service + Postgres. Host it on Render, a UIUC server, or your laptop for a demo. See the deploy guide.

PART 2·5 What gets tracked

  • Every question a student asks
  • The tutor's full answer + the mode/level used
  • Each claim and its ✓ / ? / ⚠ tag
  • Uploaded sketches & GH screenshots
  • Pasted .ghx definitions
  • The report-back observation (the trace)
  • Concept tags per answer (e.g. "graft", "flatten")
  • Class-wide "sticking points" — who's stuck on what
