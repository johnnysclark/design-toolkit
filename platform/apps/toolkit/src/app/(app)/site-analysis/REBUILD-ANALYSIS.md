## 1. Engine correctness & robustness


### [HIGH] getBoundary matches EPA_ID by LIKE '%id%' and merges every hit, fusing unrelated Superfund sites
`datasources.ts` · getBoundary, lines 192-235 · _bug_  
**Impact:** An EPA ID that is a substring of other IDs (e.g. 'IL001' inside 'IL0011') returns multiple features, and the code unions ALL their rings into one MultiPolygon, sums their GIS_AREA, and takes the combined bbox. The student then gets a boundary that is several sites stitched together, an inflated acreage, and a terrain study box derived from the wrong (possibly huge) bbox. The name-LIKE fallback (lines 201-209) has the same multi-match merge problem.  
**Fix:** Replace the substring match with an exact match mirroring getSite: `const where = `EPA_ID='${safeId}'``. Guard the empty/whitespace case first — if `!safeId.trim()` return null, so it never degrades to `LIKE '%%'` (match-all). Keep merging features that share the matched EPA_ID (operable units/feature types of the one site are meant to be unioned). For the name fallback, the LIKE is unavoidable but must be disambiguated: when the result set contains more than one distinct EPA_ID, do NOT merge across them — group by EPA_ID and select the single group whose centroid is closest to the known site centroid (pass base.centroid into getBoundary), then merge only that group's rings. This preserves correct multi-feature merging for a single site while eliminating cross-site fusion, inflated acreage, and the wrong terrain bbox.  

### [HIGH] Superfund path chains 30s+30s+30s fetch timeouts before a 40s climate fetch, exceeding the 60s function cap
`route.ts` · POST + getSite, route line 20/134/180-192; datasources fetchJson line 88, arcgis 100-103, getSite 146-189 · _robustness_  
**Impact:** In superfund mode the route runs getSite (NPL query, up to 30s) then getBoundary (up to two ArcGIS queries at 30s each, sequential) BEFORE the parallel block whose slowest member (climateYear) is 30+ seconds. On a slow-but-not-dead ArcGIS, cumulative wall time can pass maxDuration=60, so Vercel kills the function with a hard 504 instead of the graceful per-source null-degradation the design intends.  
**Fix:** Ship a total-request budget plus collapse the redundant boundary call. (1) Add an overall deadline at the top of POST, e.g. const deadline = Date.now() + 50_000, and have every fetch use timeout = min(perCallCap, deadline - Date.now()) so the function always returns degraded JSON before Vercel's 60s kill — this preserves the per-source graceful-degradation design instead of moving the cliff. (2) Tighten the ArcGIS per-call default from 30s to ~8-10s (NPL + boundary normally return in <2s; 30s is far too generous given climate alone needs ~40s). (3) Collapse getBoundary's id-match and name-fallback into a SINGLE arcgis query using an OR where-clause (EPA_ID LIKE '%id%' OR UPPER(SITE_NAME) LIKE UPPER('%name%')) to remove the second sequential round-trip. Note that full parallelization of boundary with the hard-data block is not viable as-is, because terrain's bbox is derived from the boundary result (route line 164) and flood/topo need the centroid that only the NPL query produces — so the realistic wins are the deadline budget + tighter timeouts + single boundary query, not parallelizing boundary. Optionally also tighten climate's 40s timeout to ~25s. (If the project is genuinely on Vercel Pro, raising maxDuration helps headroom but is not a fix — the unbounded sequential timeouts must still be budgeted.)  

### [HIGH] Grid-mean gap-fill is baked into slopePct, the OBJ mesh, contours, and the 3dm — not just 'exports'
`datasources.ts` · elevationGrid 417-432; terrainReadout site-analysis-tool.tsx 1083-1094; contoursDXF exporters.ts 70-111; addTerrain rhino3dm-client.ts 153-167 · _export_  
**Impact:** stats.min/max/mean correctly use only real cells (`flat`), but every DOWNSTREAM geometry consumer reads the filled grid. On a partial-coverage site (coastline, a NoData strip), the mean-filled cells create a flat plateau at the mean elevation plus a vertical step at the coverage edge: the 'Mean slope %' shown to the student is diluted toward zero, the OBJ mesh has a fake flat shelf, and marching-squares draws spurious contour rings hugging the real/fill boundary. The student sees and exports terrain that does not exist, with no visible flag beyond a `missing` count.  
**Fix:** Stop discarding the NoData mask, then fix the three real-impact paths in priority order. (1) Thread a boolean mask through: have elevationGrid return `filledMask: boolean[][]` (or keep cells null and let consumers branch) alongside the filled grid in the Topo type. (2) Slope (highest value, lowest risk): in terrainReadout, skip any interior cell whose central-difference stencil (the 4 neighbors it reads) touches a fabricated cell, so the displayed "Mean slope %" reflects only measured ground. (3) Geometry honesty in exports: in contoursDXF, skip a cell when any of its 4 corners is fabricated (kills the spurious boundary rings); in the OBJ + .3dm meshes, either hole faces whose corners are all fabricated, or emit them on a separate "Terrain (estimated)" layer / distinct vertex color so the fake flat shelf is unmistakable in Rhino. (4) Optionally shade the fabricated region on the map/preview so the student sees *where*, upgrading the existing aggregate amber count to a spatial flag. This is the already-specced [v1] "Flag fabricated terrain" follow-up; the only addition beyond the SPEC text is excluding fabricated cells from the slopePct computation, which the SPEC frames as exports-only.  

### [HIGH] floodAt reports 'No special flood hazard area' for an unmapped point and marks it as covered
`datasources.ts` · floodAt 437-467; coverage in route.ts line 200 · _data-gap_  
**Impact:** Three distinct realities collapse into two wrong buckets. (1) A point genuinely outside NFHL coverage returns mapped:true + 'No special flood hazard area' (asserting safety that was never assessed) and coverage.flood becomes true. (2) A single-point intersects-query misses a parcel that is partly in, or adjacent to, a flood zone — common for a site edge near water. (3) A timeout or transient error hits the catch and returns mapped:false ('did not return data'), which reads identically to 'this place has no flood data,' hiding that the area may in fact be a flood zone.  
**Fix:** Three changes. (1) Stop asserting safety on an empty result: the `if (!a)` (no-feature) branch must NOT return mapped:true + "No special flood hazard area" + coverage true. Treat "no polygon returned" as an unknown/unmapped state distinct from "mapped as Zone X" — e.g. return {mapped:false, note:"No NFHL flood-zone polygon at this point (area may be unmapped/outside FEMA coverage)."} or add an explicit `assessed:false` flag — so coverage.flood reflects whether FEMA was actually consulted. Confirm true Zone-X coverage with a panel/coverage query (NFHL has political-area/panel availability layers) rather than inferring "no hazard" from absence of a hit. (2) Harden the soft-failure path: fetchJson (or floodAt) must treat an ArcGIS HTTP-200 body that contains `error` or lacks a `features` array as a failure, not as "no hazard" — otherwise transient/soft FEMA errors are silently reported as a clean site. (3) Query a small buffer/envelope around the point — or, better, the actual boundary polygon (site.boundary) / bbox, both already in scope in analyze/route.ts — so a parcel partly in or adjacent to an AE zone near water is caught instead of missed by the centroid. Keep the network-error catch returning mapped:false (already correct), but give genuinely-unmapped points their own distinct state so the UI/synthesis can say "flood not assessed here" instead of "no special flood hazard area."  

### [MEDIUM] areaAcres assumes acres but GIS_AREA_UNITS is fetched and never checked
`datasources.ts` · getBoundary, lines 197/205 (outFields) and 218/231 · _data-gap_  
**Impact:** The query explicitly requests GIS_AREA_UNITS but the code never reads it; GIS_AREA is blindly summed and surfaced as `areaAcres` (and exported into boundary.geojson properties and the dossier). If the EPA layer reports any row in square meters / hectares / square miles, the acreage shown to the student is silently wrong by orders of magnitude.  
**Fix:** Stop asserting acres from an unread field. Per feature, read GIS_AREA_UNITS and normalize to acres BEFORE summing: 'Acres' → ×1; 'Square Miles' → ×640; linear units ('Miles') and 'None'/null/unknown → do NOT fold into the area total (skip them, since they aren't areal). If a site's features carry mixed areal units, convert each then sum. Most robust option: ignore GIS_AREA entirely and compute area directly from the returned WGS84 rings via a spherical/geodesic polygon-area calculation — this is unit-proof and already has the geometry in hand. Whichever path, if the resulting unit is ambiguous (linear-only or None), surface the raw value labeled with its source unit (e.g. "GIS area: 3.2 (Miles, per EPA)") rather than appending "ac", and propagate that honest label into the geojson properties + dossier instead of the bare `areaAcres`. Given the pedagogy goal of sourced/honest data, tagging the unit per source is in-spirit and cheap.  

### [MEDIUM] EPW hardcodes a Sunday start-day and assumes a full 8760-hour series with no validation
`exporters.ts` · climateEPW 176-249 (header line 199, loop 211-247, Feb-29 skip 217) · _export_  
**Impact:** The start day-of-week is hardcoded 'Sunday', which is only true for 2023; if CLIMATE_YEAR is ever changed (e.g. to a normal or to 2024), EnergyPlus/Ladybug schedules will align weekends/holidays to the wrong days, and the leap-day skip leaves the row-year field at the leap year while still claiming Sunday/8760. There is also no guarantee the file is exactly 8760 rows: it emits one row per returned hour, so any short or gappy Open-Meteo response (or a future leap year sourced) yields a file EnergyPlus rejects, with no padding or length check despite the 'strictly 8760 rows' comment.  
**Fix:** Decouple the EPW from the hardcoded year and validate length, with these concrete changes in climateEPW (exporters.ts): (1) Compute the DATA PERIODS start day-of-week from the actual Jan 1 of climate.year instead of the literal "Sunday": const dow = ["Sunday","Monday","Tuesday","Wednesday","Thursday","Friday","Saturday"][new Date(Date.UTC(climate.year,0,1)).getUTCDay()]; and interpolate `...Data,${dow},1/1,12/31`. (2) After the loop, enforce exactly 8760 rows: if (rows.length !== 8760) throw a clear, surfaced error (e.g. `EPW generation aborted: got ${rows.length} hours (expected 8760) — incomplete Open-Meteo response`). Because this is the artifact the student is told to verify, fail loud rather than silently emit a malformed file; padding with missing-code rows is acceptable only if you also keep the dates contiguous, but rejecting is safer here. (3) Keep the Feb-29 skip (it correctly handles leap years), but note in the COMMENTS line when a leap day was dropped so the year field and 8760 count are not silently inconsistent. (4) Optionally guard upstream in climateYear by asserting h.time.length is 8760 (or 8784 for a leap year) so the failure is caught before export. These remove the 2023 coupling entirely and make the "strictly 8760 rows" comment actually enforced.  

### [MEDIUM] prevailingWind defaults to 'N' (fraction 0) when there is no wind data
`route.ts` · summarizeClimate 83-100; mirrored in rhino3dm-client.ts addLabels 300-302 · _ai_  
**Impact:** If wind data is missing or every hour is calm, all `totals` are 0; `Math.max(...zeros)` is 0 and `indexOf(0)` returns index 0, so the tool confidently reports prevailing wind 'N' with fraction 0 — a fabricated cardinal direction baked into the summary, the 3dm 'Prevailing wind: N' label, and the dossier. This contradicts the tool's stated pedagogy of being honest about coverage.  
**Fix:** Treat all-zero totals as "no prevailing direction" and surface it honestly. In summarizeClimate (route.ts ~85-100): compute maxTotal = Math.max(...totals) and, when maxTotal === 0 (equivalently n===0 or calmFraction===1), set prevailingWind to null instead of {dir:'N', fraction:0}. Widen the type (types.ts:48) to `prevailingWind: { dir: string; fraction: number } | null` and update the three consumers to render "insufficient wind data" when null: the UI Stat (site-analysis-tool.tsx:567), the markdown dossier (exports.ts:156), and the 3dm label. In rhino3dm-client.ts addLabels (299-314), gate the "Prevailing wind" text dot on the same `climate.windSpeed?.length` (and non-zero maxTotal) condition already used for the wind-rose geometry at line 124, or render "Prevailing wind: insufficient data" — so the label and geometry stay consistent. Low-to-moderate priority; bundle with the broader coverage-honesty pass.  

### [MEDIUM] Sun-path year is hardcoded 2023 in two places, with a duplicated local→UTC conversion and a single fixed tz offset
`geo.ts` · sunPaths 111-131 (year line 117, tz conversion 122-123); duplicate localUTC in rhino3dm-client.ts 188-192 · _robustness_  
**Impact:** The solstice/equinox year is hardcoded independently of CLIMATE_YEAR in both the chart engine and the 3dm engine, so changing the climate year (e.g. to a normal) silently leaves sun geometry on 2023 dates — two sources of truth that can drift. The local→UTC conversion is copy-pasted between geo.ts and rhino3dm-client.ts (a fix to one won't reach the other). tzOffsetHours is a single fixed value from utc_offset_seconds, so the diagram ignores DST: a summer-solstice 'noon' arc is plotted against standard-time clock hours, shifting hour markers by up to an hour relative to a student's civil clock.

Hoist a single `localUTC` helper into geo.ts and import it into rhino3dm-client.ts so the local→UTC math lives in one place. For DST, the cheapest honest fix for a teaching tool is to document/label that arcs are plotted in standard (non-DST) local solar time and mark hour ticks accordingly; a fuller fix computes the per-date civil offset. Note in the recommendation that the year-coupling is largely cosmetic (solstice/equinox geometry barely moves between years and the dates are already hardcoded), so prioritize the single-source-of-truth + DST labeling over expecting a visible geometry change.  

### [LOW] solarPosition omits atmospheric refraction; altitude>0 cutoff clips sunrise/sunset slightly
`geo.ts` · solarPosition 94-105; sunPaths altitude filter line 125; rhino3dm-client.ts 214/223 · _robustness_  
**Impact:** The altitude is the true (geometric) elevation with no refraction term, and the diagram includes points only when altitude > 0. The real sun is visible at geometric altitude ≈ -0.83° at sunrise/sunset, so the sun-path arcs start ~a few minutes late and end early, and the lowest-sun azimuths (the ones architects care about for early/late shading and glare) are dropped. The peakAltitude summary is unaffected, but the horizon end-caps are biased.  
**Fix:** Ship as a low-priority correctness/clarity fix. Preferred: add the standard atmospheric-refraction correction to solarPosition's returned altitude (apparent altitude) and KEEP the `> 0` cutoff in all three spots (geo.ts 125, rhino3dm-client.ts 214/223). That fixes the horizon endpoints AND avoids plotting below-horizon points. Do NOT simply lower the cutoff to `> -0.833` without clamping — the SVG polar chart maps altitude to radius, so negative-altitude points would render outside the horizon rim. Lowest-effort acceptable alternative, consistent with the tool's 'honest, sourced data' pedagogy: leave the math as-is but document at solarPosition that altitude is geometric/true (no refraction) and that arcs are clipped at the geometric horizon, so the ~few-minute end-cap bias is understood. Either way, also note the dominant endpoint error today is the 30-min (sunPaths) / 15-min (rhino) sampling granularity, not refraction — if endpoint fidelity matters, tighten sampling near the horizon before worrying about refraction.

## 2. Data richness — what to display

`datasources.ts` · climateYear, lines 500-501 · _data-gap_  
**Impact:** Rainfall drives stormwater sizing, roof/drainage form, snow load, and landscape strategy — it is arguably the most fundamental climate variable for a building, and the tool reports none of it. The generated EPW also lacks liquid precipitation and snow, so Ladybug stormwater/snow studies are impossible from it.  
**Fix:** Add `precipitation,rain,snowfall,snow_depth` (optionally `et0_fao_evapotranspiration`) to the same single Open-Meteo hourly= request at datasources.ts:500-501 — free, one round-trip, ERA5 archive supports all of them. Extend the ClimateRaw interface and the return map with the parallel arrays (e.g. precip[], rain[], snowfall[], snowDepth[]). ### [HIGH] Dew point, DNI/DHI, sky cover and pressure are fetched but only routed to the EPW — never derived or displayed
`route.ts` · summarizeClimate + climateExtras, lines 41-113 · _data-gap_  
**Impact:** The direct/diffuse split (DNI vs DHI) governs shading, glare and daylight design; sky cover tells overcast vs clear-sky character; dew point drives condensation, mold and evaporative-cooling potential. The tool already paid the network cost to fetch all of them and shows none — students never see the data that most informs envelope and daylighting.  
**Fix:** Ship it, but derive server-side rather than client-side so the values also flow into the dossier .json/.md and stay single-sourced. Add to climateExtras(climate) three pure reductions over the existing hourly arrays: (1) diffuseFraction = Σdhi / Σghi (sum over daylight hours where ghi>0; both are horizontal-plane irradiance, so this is the valid direct/diffuse split for shading/glare/daylight) — surface as a 'Diffuse fraction' Stat with a plain-language 'overcast vs crisp-sun' note; (2) meanCloudCover = mean(cloud) and clearSkyHours = count(cloud < ~20%) from cloud_cover; (3) muggyHours = count(dewC > 16) for a condensation/evaporative-cooling read (note the 16°C threshold is the 'sticky/humid' line; 18°C is 'oppressive' — pick one and label it). Add the matching Stats to the Macro Climate card and pull dni in too if you want a 'beam-dominant hours' count for solar-control framing. Two caveats to honor the tool's own pedagogy: tag all of these to the same single-year-2023 ERA5 caveat the rest of the climate panel already carries (these are one synthetic year, not a normal), and keep the Stats text black per the studio's all-text-black rule rather than the neutral-400/500 used elsewhere in the charts. Trivial effort; no new network calls.  

### [HIGH] Solar-by-surface-orientation is uncomputed despite solarPosition() + DNI/DHI/GHI all being present
`route.ts` · climateExtras lines 41-79; geo.ts solarPosition lines 48-106 · _data-gap_  
**Impact:** Incident solar on each facade orientation is the single most actionable passive-design metric (where to put glazing, where heat gain is a liability) and every ingredient is already in memory — yet the tool stops at one horizontal number.  
**Fix:** Ship it, computed server-side in climateExtras (the loop already walks every hour, so it costs one extra accumulation and feeds both UI and dossier with no client re-derivation). For each hour with altitude>0, compute cos(incidence) for each vertical facade S/E/W/N as cos(alt)·cos(azSun − azSurface) clamped to ≥0; accumulate beam = DNI·cos(incidence) + isotropic diffuse = DHI·0.5 (+ optional ground-reflected 0.2·GHI·0.5) into Wh/m², divide by 1000 for kWh/m²·yr per orientation. For the roof bar, prefer tilt-at-latitude (PV-actionable) rather than flat horizontal, which would just duplicate the existing GHI number. Surface a 5-bar mini-chart next to Sun path in the Macro Climate card and add the per-orientation totals to the dossier .md/.json. Two non-negotiables for this repo: (1) tag the numbers as a single 2023 year (inherits the known hardcoded-year caveat — not a climate normal); (2) the new chart must use neutral-900 labels (all-text-black rule) and ship an ARIA/data-table fallback (the existing charts-are-SVG-only gap). Keep it in the public D0 pass — it is pure hard-data math, no model, no key.

`route.ts` · summarizeClimate.annual, lines 103-110 · _data-gap_  
**Impact:** Degree-days summarize demand but not livability; a student cannot see 'comfortable X% of the year, too humid Y%, passive ventilation viable Z hours' — the bridge from raw climate to design intent. ### [HIGH] Per-cell slope direction (aspect) is computed then thrown away — only gradient magnitude is kept
`site-analysis-tool.tsx` · terrainReadout, lines 1086-1094 · _data-gap_  
**Impact:** Aspect determines solar exposure of the ground, drainage direction, and cold-air pooling — a south-facing 8% slope and a north-facing 8% slope are completely different sites, but the tool reports them identically as 'Mean slope ~8%'.  
**Fix:** Ship it as a low-cost enhancement to the Micro Terrain card. In the same interior-cell loop, also accumulate the gradient vector: sgx += gx; sgy += gy. After the loop, compute the dominant ASPECT as the downhill-facing direction (negate the uphill gradient): bearingDeg = (Math.atan2(-sgx, -sgy) * 180/Math.PI + 360) % 360, matching the repo's azimuth convention (0=N, 90=E, clockwise, as defined in lib/site-analysis/geo.ts solarPosition). Map bearingDeg to an 8- or 16-point compass label (N/NE/E/.../NW) and expose it from terrainReadout (e.g. aspectLabel, aspectDeg). Add a Stat to the terrain grid: label "Dominant aspect", value `${aspectLabel}-facing`, hint noting it is derived from the coarse elevation grid and drives solar exposure, drainage, and cold-air pooling. Guard against meaningless bearings: if the resultant vector magnitude is negligible relative to mean slope (bowl/dome or near-flat terrain), report "no dominant aspect (flat/mixed)" instead of a noisy direction. Summing raw gradient vectors is appropriate here because near-flat cells contribute near-zero and naturally down-weight themselves. Optional follow-up (separate change): offer an aspect color ramp as an alternate to the elevation ramp on the SiteMap dots. Per the studio all-text-black rule, keep the new Stat/label text black, not grey.  

### [HIGH] Terrain reduced to four scalars — no slope distribution, buildable-area %, cut/fill, or high/low-point siting

## Research — free/keyless data layers available


### Free, keyless US geospatial data services to extend the Surveyor site-analysis tool (beyond EPA NPL, USGS 3DEP, FEMA NFHL, Open-Meteo, OSM Nominatim)

I verified 15 free US geospatial services an architecture studio could wire into Surveyor server-side, each beyond what it already uses. The strongest, easiest wins are ArcGIS REST 'query'/'export'/'identify' and ImageServer 'identify'/'getSamples' endpoints (no key, single request, JSON/GeoJSON out): USDA SSURGO soils (SDA POST T-SQL), USGS StreamStats watershed delineation, MRLC/NLCD land cover + % impervious + % tree canopy, USGS The National Map (contours/NHD hydro/structures/transportation)

- **USDA SSURGO soils via Soil Data Access (SDA)** (yes key · US (CONUS + most of AK/HI/PR/territories; a few unmapped areas fall back to STATSGO) · moderate) — Detailed soil survey: drainage class, hydrologic soil group (A-D), depth to water table/bedrock, available water capacity, %clay/sand, Ksat, slope, farmland class, shrink-swell, frost action, and bearing/engineering limitations per soil map unit. → _Soil is foundational to siting: drainage/HSG drives stormwater + infiltration/rain-garden design, depth-to-water-table + shrink-swell + bearing inform foundation type, slope/erosion inform grading. Pairs naturally with Surveyor's terrain/flood read._ · `https://sdmdataaccess.nrcs.usda.gov/webservicehelp.aspx`
- **USGS StreamStats watershed delineation + basin characteristics** (yes key · US, state-by-state (must pass the state 'rcode'; a few states not yet enabled) · moderate) — Delineate the drainage basin upstream of a point (GeoJSON polygon), plus basin characteristics (drainage area, % impervious, % forest, mean annual precip, slope) and regression-based peak/flow statistics. → _Shows the actual contributing watershed a site sits in and how much water drains toward/through it — directly supports stormwater strategy, bioswale sizing, and reading the site as part of a hydrologic system rather than an isolated parcel._ · `https://streamstats.usgs.gov/docs/streamstatsservices/`
- **MRLC/USGS NLCD land cover + % impervious + % tree canopy** (yes key · CONUS (land cover/impervious/canopy); AK/HI/PR partial depending on product · moderate) — National Land Cover Database: 16-class land cover, fractional/percent impervious surface, and percent tree-canopy cover rasters (30 m), multiple epochs. → _Quantifies the site's context in one number each: developed/forest/wetland class, % impervious (runoff + urban-heat proxy), % canopy (shade, cooling, microclimate). Canopy + impervious are exactly the metrics a studio uses to argue for tree retention or green infrastructure._ · `https://www.mrlc.gov/data-services-page`
- **USGS The National Map — contours, NHD hydro, structures, transportation** (yes key · US · easy) — Vector reference layers via ArcGIS MapServer export/query/identify: USGS contour lines (large + 50/100 ft), NHD streams/waterbodies/wetlands, National Structures (schools, hospitals, fire/police, post offices), and transportation (roads, rail, trails, airports). → _Gives ready-made site-context vectors: existing published contours (a sanity check against Surveyor's 3DEP-generated contour DXF), nearby water features, the road/rail edges of a parcel, and civic structures for program/adjacency analysis — all directly exportable to a base drawing._ · `https://www.usgs.gov/faqs/where-can-i-find-a-list-urls-national-map-services`
- **EPA EJScreen — environmental burden + EJ indices** (yes key · US · easy) — Block-group environmental indicators (modeled PM2.5, ozone, diesel PM, air toxics cancer/respiratory risk, traffic proximity, lead-paint, proximity to RMP/TSDF/Superfund/wastewater), demographic index, and combined EJ indices with percentiles. → _A one-call environmental-justice + pollution-burden read on the site's community — gives the studio modeled air-quality numbers (a keyless substitute for AirNow) and a defensible EJ framing for the project, complementing Surveyor's contamination pass._ · `https://ejscreen.epa.gov/arcgis/rest/services/ejscreen`
- **EPA EnviroAtlas — ecosystem services + urban greenspace/heat** (yes key · US national (coarse) + ~30 EnviroAtlas 'Communities' at 1 m resolution · moderate) — 500+ layers: 1 m urban land cover for select communities, tree-canopy & greenspace near homes/schools, urban heat/imperviousness, floodplain & riparian buffers, walkability, pollinator and water-quality ecosystem-service metrics. → _Ecosystem-services lens an architecture studio increasingly needs: greenspace access equity, urban-heat exposure, and 1 m land cover for fine-grained site context in covered cities — strong for sustainability/landscape narratives._ · `https://www.epa.gov/enviroatlas/enviroatlas-web-services`
- **USGS ASCE 7 / NEHRP seismic design parameters** (yes key · US + territories (per the chosen reference document's model extent) · easy) — Code seismic design values at a lat/lon for a given Risk Category + Site Class: Ss, S1, SMS, SM1, SDS, SD1, Seismic Design Category (SDC), TL, and design response spectrum points. → _Turns a location into the actual structural seismic design category and base code parameters — lets the studio state, early, what lateral system / structural ambition the site demands. Single request, perfectly fits Surveyor's 'verifiable hard data' card._ · `https://earthquake.usgs.gov/ws/designmaps/asce7-22.html`
- **USFS Wildfire Hazard Potential + Wildfire Risk to Communities** (yes key · US (conterminous; some products incl. AK/HI) · easy) — Wildfire Hazard Potential (WHP) index, annual burn probability, conditional/total risk to potential structures, and flame-length exceedance probabilities, as ImageServer rasters. → _Wildfire exposure is now a primary siting + envelope/material driver in much of the West and WUI — WHP and risk-to-structures give a defensible hazard read for defensible-space, ember-resistant detailing, and access planning._ · `https://apps.fs.usda.gov/fsgisx01/rest/services/RDW_Wildfire/RMRS_WRC_WildfireHazardPotential/ImageServer`
- **NOAA Digital Coast Sea-Level-Rise inundation** (yes key · US coastal (including territories where SLR Viewer covers them) · easy) — Sea-level-rise inundation extent + depth layers from 0–10 ft above MHHW, plus marsh-migration and flood-frequency (low-lying) layers. → _Adds a FUTURE coastal-flood dimension that FEMA NFHL (present-day) misses — lets a coastal studio test the site against 1–6 ft SLR scenarios for finished-floor elevation, resilience, and managed-retreat narratives._ · `https://coast.noaa.gov/arcgis/rest/services/dc_slr`
- **NOAA NCEI U.S. Climate Normals (1991–2020)** (yes key · US (~15,000 stations) · moderate) — Official 30-year normals per station: monthly/daily/hourly + annual/seasonal normal temperature, precipitation, snowfall, and heating/cooling degree days. → _Directly fixes Surveyor's biggest known data flaw — climate hardcoded to a single 2023 year. A true 1991–2020 normal (incl. HDD/CDD) for the nearest station gives a defensible baseline for passive-design and energy reasoning, and an honest contrast to the single-year ERA5 used for the EPW._ · `https://www.ncei.noaa.gov/access/us-climate-normals/`
- **U.S. Census TIGERweb + Geocoder + ACS demographics** (yes key · US · easy) — Census geographies (tract/block-group/place/county boundaries) via TIGERweb ArcGIS, address↔lat/lon + FIPS via the Geocoder, and ACS 5-year socioeconomic data (population, density, age, income, housing, tenure, commute) via the Data API. → _Grounds a project in its community: density, demographics, housing, and income for the surrounding tract/block group inform program, unit mix, and equity framing — the social-context layer Surveyor currently lacks._ · `https://www.census.gov/data/developers/data-sets/TIGERweb-map-service.html`
- **USGS PAD-US protected areas + public lands / GAP** (yes key · US · easy) — Protected Areas Database: parks, open space, conservation easements, public land ownership/manager, designation type, and GAP status (1–4 biodiversity-protection intent). → _Reveals nearby protected/conserved/public lands and ownership around a site — supports open-space adjacency, access-to-nature, and land-status due-diligence in early siting._ · `https://www.usgs.gov/programs/gap-analysis-project/science/pad-us-web-services`
- **NWS api.weather.gov — gridded forecast + observations** (yes key · US + territories · easy) — Per-2.5 km gridpoint forecast (temp, wind, sky cover, precip prob, dewpoint, RH), hourly forecast, and nearest-station current observations — fully open JSON (GeoJSON/JSON-LD). → _A genuinely keyless, no-signup live-weather + short-term forecast feed for the exact site — useful for a 'current conditions' card and as a sanity check on the historical climate read, with no token to manage._ · `https://weather-gov.github.io/api/general-faqs`
- **USGS Water Services (NWIS) — streamflow + groundwater levels** (yes key · US · easy) — Real-time + historical instantaneous/daily values for nearby monitoring sites: streamflow, gage height, water temperature, and — notably — groundwater (water-table) levels from observation wells. → _Nearest stream-gage flow/stage and especially groundwater depth give a real, measured read on the water table and surface-water behavior near a site — high-value for foundations, basements, dewatering, and drainage that Surveyor's flood point-query alone can't supply._ · `https://waterservices.usgs.gov/docs/instantaneous-values/instantaneous-values-details/`
- **USDA NASS CropScape — Cropland Data Layer (CDL)** (yes key · CONUS (30 m 2008–2023; 10 m from 2024) · moderate) — Annual crop-specific land cover (corn/soy/orchard/fallow/developed/etc.), point value (GetCDLValue), clipped image/stats, and year-over-year change. → _For rural/peri-urban or agrarian sites, identifies exact crop/agricultural land use and how it has changed — useful context for adaptive-reuse, farm/landscape, and land-conversion narratives that generic land cover (NLCD) can't distinguish._ · `https://nassgeodata.gmu.edu/CropScape/`


### Free/keyless global data sources to extend Surveyor (terrain, flood, land cover, buildings, climate) beyond its US-only USGS 3DEP + FEMA + single-year ERA5 coverage

Surveyor's three US-only or thin spots each have a genuinely free, mostly keyless global replacement that fits its existing server-route + ArcGIS-export pattern. ELEVATION: the cleanest drop-in for USGS 3DEP getSamples is Open-Meteo's Elevation API (keyless, GLO-90 90m, 100 coords/request, same lat/lng-array -> elevation-array shape); for a continuous mesh worldwide, AWS Terrain Tiles (keyless S3 terrarium PNG / GeoTIFF, ~30m, decode elev=(R*256+G+B/256)-32768) or Copernicus DEM GLO-30 30m COGs 

- **Open-Meteo Elevation API** (yes key · global · easy) — Point elevation lookup worldwide (Copernicus GLO-90, 90m). Returns {"elevation":[...]} array; accepts up to 100 lat/lng pairs per request. → _Cleanest drop-in replacement for the US-only USGS 3DEP getSamples: same shape (send a grid of lat/lng, get an elevation array back), so non-US sites stop returning null and still feed the .obj mesh / .dxf contours / .3dm terrain exports._ · `https://open-meteo.com/en/docs/elevation-api`
- **AWS Terrain Tiles (Mapzen terrarium / GeoTIFF / Skadi)** (yes key · global · moderate) — Global tiled elevation as terrarium-encoded PNG, GeoTIFF, or Skadi HGT on a public S3 bucket; ~10-30m near coasts (3DEP/SRTM/GMTED blend) up to z15. → _Lets the terrain mesh be continuous and seamless worldwide: fetch one or a few tiles around the site, decode pixels to a height grid, and build the .obj/.dxf/.3dm without per-point API round-trips. Same z/x/y pattern Surveyor already uses for basemaps._ · `https://registry.opendata.aws/terrain-tiles/`
- **Copernicus DEM GLO-30 via Microsoft Planetary Computer** (yes key · global · moderate) — True 30m global DSM as Cloud-Optimized GeoTIFFs, discoverable through a STAC API; read just the window covering the site. → _Authoritative 30m DSM (3x finer than Open-Meteo's 90m) for a crisp non-US terrain mesh + contours, without any account — good when the studio wants verifiable, named-source elevation rather than blended tiles._ · `https://planetarycomputer.microsoft.com/dataset/cop-dem-glo-30`
- **OpenTopography Global DEM API** (no key · global · moderate) — Server-side DEM raster (COP30/COP90/SRTMGL1/NASADEM, ~30m) clipped to a bbox, returned as GeoTIFF or AAIGrid in one request. → _One call returns a ready-to-contour DEM clipped to the exact site bbox (ideal for the .dxf contour + .obj mesh pipeline) — but it requires a free API key, so it violates Surveyor's keyless-server-side rule._ · `https://portal.opentopography.org/apidocs/`
- **JRC Global Surface Water** (yes key · global · moderate) — 30m global water occurrence / seasonality / recurrence / change, 1984-2021, from Landsat. Keyless cached WMTS tiles for overlay, plus COGs on Planetary Computer for a sampled value. → _Global stand-in for the US-only USGS NHD water layer: shows permanent vs seasonal water and how water extent has changed around the site — directly useful for a site-water/hydrology read and as a low-cost flood proxy._ · `https://global-surface-water.appspot.com/download`
- **JRC Global River Flood Hazard Maps v2** (yes key · global · hard) — Gridded river-flood inundation depth at ~3 arc-sec (~90m) for return periods rp10, rp20, rp50, rp100, rp200, rp500, as downloadable GeoTIFF tiles. → _The closest worldwide analog to FEMA NFHL's return-period flood zones: gives a real '1-in-100-year inundation depth at this site' value for non-US locations, which is exactly the flood read Surveyor currently can only do in the US._ · `https://data.jrc.ec.europa.eu/dataset/jrc-floods-floodmapgl_rp50y-tif`
- **WRI Aqueduct Floods Hazard Maps v2** (yes key · global · hard) — Riverine AND coastal flood inundation, ~1km, for current conditions plus future projections (2030/2050/2080) under climate+socioeconomic scenarios. → _Adds two things FEMA/JRC-river don't: coastal flooding and forward-looking climate-scenario flood (a strong teaching hook — 'this coast's 1-in-100 flood in 2050 under a warming scenario'). Pairs naturally with Surveyor's verified/plausible tagging._ · `https://www.wri.org/data/aqueduct-floods-hazard-maps`
- **ESA WorldCover 10m (Terrascope WMS + AWS COGs)** (yes key · global · easy) — Global 10m land cover, 11 classes (tree cover, built-up, cropland, water, wetland, etc.) for 2020 and 2021. Keyless WMS GetMap overlay + GetFeatureInfo for the class at a point; also no-sign AWS COG tiles. → _A whole new, genuinely global site-context card Surveyor lacks: what's actually on and around the ground (forest vs built-up vs crop vs water). 10m is fine enough to characterize a parcel and its surroundings, and it complements terrain/flood for a fuller site read._ · `https://docs.terrascope.be/Developers/WebServices/OGC/WMS.html`
- **Google Dynamic World (near-real-time 10m land cover)** (no key · global · hard) — 10m, 9-class land cover with per-pixel class probabilities, updated every Sentinel-2 pass (near real time) rather than annual. → _Richest, most current land-cover option (probabilities + frequent updates) — but only worth noting as an upgrade path because it is Earth-Engine-gated, so it cannot be used keyless server-side today._ · `https://developers.google.com/earth-engine/datasets/catalog/GOOGLE_DYNAMICWORLD_V1`
- **OSM Overpass API** (yes key · global · easy) — On-demand vector query of OpenStreetMap worldwide: building footprints, roads, water bodies, landuse, etc., returned as JSON/XML you can turn into GeoJSON. → _Immediate worldwide context for the boundary/context cards and the .dxf context export: real building footprints, street network, and water around the site — the global complement to the US-centric layers, with vector geometry that drops straight into Rhino-ready exports._ · `https://wiki.openstreetmap.org/wiki/Overpass_API`

### Making Surveyor's climate read richer and more credible for architects, keyless — beyond a single ERA5 year (2023)

Surveyor currently builds its wind rose, sun path, monthly temp/RH and EPW from ONE Open-Meteo ERA5 year (2023), which is not a climate normal and silently hides interannual variability. The biggest, cheapest credibility win is to keep the same keyless Open-Meteo Historical (ERA5) endpoint but (a) request a multi-decade window (e.g. 1991-2020, the WMO 30-yr normal period) and compute monthly means + percentiles instead of one year, and (b) pull the extra hourly/daily variables that are already e

- **Multi-year ERA5 normal (replace single 2023 with a 30-yr window)** (yes key · global (reanalysis grid, any lat/lon incl. non-US — unlike Surveyor's US-only terrain/flood) · moderate) — Same Open-Meteo Historical/ERA5 archive endpoint, but request a long window (e.g. 1991-01-01..2020-12-31, the WMO normal period) in ONE call and compute monthly means + percentiles + interannual spread, instead of one hardcoded year. Lets you build a real TMY-like 'typical' (ISO 15927-4 / Sandia TMY selects the most-representative each calendar month) rather than whatever weather 2023 happened to have. → _A single year hides drought/heatwave/cold years; a 30-yr normal is what architects, ASHRAE 169 and EPW standards actually expect, and percentiles unlock real design conditions (see ASHRAE entry). - **Unused hourly/daily variables already exposed by ERA5** (yes key · global · easy) — Add the exact parameters Surveyor isn't pulling: soil_temperature_0_to_7cm…100_to_255cm, soil_moisture_0_to_7cm…, snowfall / snow_depth, wind_gusts_10m (+ daily wind_gusts_10m_max), vapour_pressure_deficit, and daily et0_fao_evapotranspiration, shortwave_radiation_sum, temperature_2m_max/min. - **Compute ASHRAE 169 heating/cooling design conditions from percentiles** (yes key · global (computed from Open-Meteo data); ashrae-meteo.info itself is a manual lookup, not an API · moderate) — From the multi-year hourly temp (and dew point/RH) compute heating 99.6%/99% and cooling 0.4%/1%/2% design dry-bulb (and mean-coincident wet-bulb), i.e. the percentile temperatures HVAC and passive sizing use. - **Thermal comfort hours: UTCI + PMV/PPD + ASHRAE-55/EN-16798 adaptive** (yes key · global (computed from temp + RH + wind + radiation, all in the ERA5 pull) · hard) — Per-hour comfort classification over the full year: UTCI (outdoor heat/cold stress), PMV/PPD (ISO 7730), and the adaptive comfort band (ASHRAE 55 / EN 16798-1) using running-mean outdoor temp — then % hours comfortable / too hot / too cold, by month and hour-of-day (comfort heatmap). → _Moves the climate read from raw weather to 'how many hours is this site naturally comfortable, and when does it fail?' — the question that justifies passive vs active strategy.
